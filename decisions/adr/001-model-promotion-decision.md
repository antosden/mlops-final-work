# ADR-001: Выбор улучшенного serving-контура на основе latency-метрик

## Status

Accepted

## Context

Для ML-системы дедупликации профилей важно обеспечить стабильное и низкое время отклика backend API.

В рамках Metrics Driven Development были сравнены два набора latency-метрик:

- latency существующей системы;
- latency улучшенной системы.

Цель анализа — проверить, действительно ли улучшенная архитектура снижает время отклика статистически значимо, а не выглядит лучше только по визуализации.

## Metrics

Основная метрика:

- latency, seconds.

Дополнительные метрики:

- mean latency;
- p95 latency;
- p99 latency.

Mean latency показывает среднее время отклика системы.  
p95 и p99 latency позволяют оценить поведение системы на медленных запросах и хвостах распределения.

## Hypotheses

H0: среднее время отклика улучшенной системы не меньше среднего времени отклика существующей системы.

H0: mean_improved >= mean_existing

H1: среднее время отклика улучшенной системы меньше среднего времени отклика существующей системы.

Формально:

H1: mean_improved < mean_existing

## Statistical test

Для проверки гипотезы был выбран Welch's t-test.

Причины выбора теста:

- сравниваются две независимые выборки latency;
- анализируется различие средних значений времени отклика;
- Welch's t-test не требует равенства дисперсий двух выборок.

Уровень значимости:

alpha = 0.05

## Results

Среднее время отклика существующей системы составляет примерно 3.5 секунды.

Среднее время отклика улучшенной системы составляет примерно 2.0 секунды.

Снижение среднего latency составляет примерно 42.8%.

По результатам Welch's t-test полученное p-value меньше 0.05.

Следовательно, H0 отклоняется.

Это означает, что улучшенная система статистически значимо снижает время отклика по сравнению с существующей системой.

## Decision

На основе Metrics Driven Development принимается архитектурное решение использовать улучшенный serving-контур, так как он статистически значимо снижает latency.

В рамках проекта это означает:

- использовать FastAPI backend как отдельный model serving service;
- разворачивать backend service в Docker;
- деплоить сервис на Google Compute Engine VM через GitHub Actions;
- контролировать latency как ключевую техническую SLI;
- использовать mean latency, p95 latency и p99 latency для дальнейшего мониторинга системы.

Итоговое архитектурное решение:

Принять улучшенную архитектуру serving-сервиса как целевую, так как статистический анализ показал значимое снижение latency.

## Consequences

Принятое решение снижает среднее время отклика примерно с 3.5 до 2.0 секунд.

Это улучшает пользовательский опыт, снижает риск таймаутов и повышает устойчивость backend-сервиса.

В дальнейшем при изменении архитектуры системы решения должны приниматься через MDD:

- сбор метрик;
- визуализация распределений;
- формулировка H0 и H1;
- выбор уровня значимости;
- проведение статистического теста;
- фиксация итогового архитектурного решения в ADR.
